1) Goals & KPIs (lock direction)

  • Stability KPI: long-run recover_event, human_intervention, drop
  • Quality KPI: Replayable trajectories (same command stream ‚Üí same outcome)
  • Throughput KPI: valid episodes per hour (filter non-replayable ‚Äúwaste data‚Äù)
  • Safety KPI: over-current / over-temp / soft-limit trips

Key intuition: DexHand wins by long-horizon stability + fast recovery + replayable data flywheel, not single-episode success.

2) Overall architecture

Three things must hold simultaneously: (i) layered sensing, (ii) multi-rate loop decomposition, (iii) RM-in-loop data flywheel. 

Level-C  (5‚Äì15Hz):   semantic policy / primitive switching
Level-B  (60‚Äì200Hz): tactile coordination (slip control, grip scheduling, micro-geometry)
Level-A  (500‚Äì1000Hz): safety reflex (current limit, jam detect, backoff)

Side loop: timestamp ‚Üí alignment ‚Üí episode segmentation ‚Üí RM scoring ‚Üí hard cases ‚Üí re-train

Design principles: timestamp-at-source, multi-rate loops, contact observability (current/effort/tactile), maintainability first, start with MVS.

3) Sensing stack (MVS ‚Üí Pro) + best placement

Level-0 (MVS): rgb_static + rgb_wrist + joint q/qdot + motor_current/joint_effort + software events (contact_flag, safety_trip).

Level-1 (tactile): the best KPI-balanced placement is ‚Äú2 + 1 + 1‚Äù:

  • 2 fingertips with high-rate shear/pressure arrays (100‚Äì200Hz): thumb + index (main slip-control loop)
  • 1 fingertip with high-information geometry (visuotactile 30‚Äì60Hz): thumb or index (contact point/area/edge/micro-slip)
  • 1 wrist 6D F/T (‚â•200Hz): push/pull/twist/pry causality (external wrench / constraint reaction)

Why not all visuotactile: cable bandwidth + durability + calibration burden scales poorly and can reduce replayability/uptime.

4) Multi-rate loops (A/B/C)

  • Level-A (500‚Äì1000Hz): safety reflex (jam/stall detect, current limiting, backoff)
  • Level-B (60‚Äì200Hz): slip control + grip scheduling + micro contact geometry adjustment
  • Level-C (5‚Äì15Hz): choose force primitives + switch recovery paths; set limits/params for A/B

5) Humanoid view: hand as ‚Äúphysical causality interface‚Äù

For humanoids, the hand is not ‚Äújust grasp‚Äù. It converts intent into physical causes: force/torque/contact constraints. A minimal task-semantics interface is the 6D wrench: 

w = [Fx, Fy, Fz, Tx, Ty, Tz]^T
push/pull: tangential force (Fx,Fy)
press: normal force (Fz)
twist/pry: torque (T*)

Slip is fundamentally about contact mode + friction budget. A simple boundary is ||f_t|| ‚â§ Œº f_n. ‚ÄúPush vs pull‚Äù is really choosing a force primitive under constraints.

6) RM-in-loop data flywheel

RM is a ‚Äúquality judge‚Äù that enables automatic segmentation, hard-case mining, and recovery triggering. Recommended multi-head outputs: progress, quality, recoverability, failure_mode, plus primitive/contact_mode/slip_risk for physical causality.

7) Time sync & data pipeline

  • PTP + timestamp-at-source
  • Ring-buffer alignment (vision anchor)
  • Log both command stream and execution stream (cmd vs measured) for replayability diagnosis

8) Calibration, health & maintainability

  • Version calibration assets (calib_version, calib_hash)
  • Visuotactile is a consumable: baseline re-zero per shift; mark recalib_required on skin replacement
  • Track sensor_health (drift, aging, cable dropouts)

9) Minimal build + roadmap (toward beyond-human)

Start: MVS + replay validation + rule-based RM. Then add 2 shear/pressure fingertips (Level-B), then 1 visuotactile fingertip + 1 wrist F/T. ‚ÄúBeyond-human‚Äù is achieved by bandwidth + dynamic range + explainability + self-labeling flywheel + long-run robustness.
